# TPC-H Generated Artifacts

TPC-H query SQL, fixed-substitution parameter stubs, workload manifests, and PostgreSQL tables are generated by the DuckDB-backed `tpch-generator` Compose service.

Generated artifacts are intentionally not committed. They are written under:

```text
benchmark-data/tpch/sf<scale-factor>/
```

The benchmark engine remains PostgreSQL. DuckDB is used only to generate reproducible local TPC-H pilot artifacts through its `dbgen` and `tpch_queries()` functions and to load generated rows into PostgreSQL through its `postgres` extension.

Two manifests are generated for each scale factor:

- `tpch-duckdb-corpus.json`: baseline-only calibration for all 22 fixed DuckDB TPC-H query instances.
- `tpch-local-llm-corpus.json`: local LLM candidate search for all 22 fixed DuckDB TPC-H query instances.
- `tpch-parameterized-baseline-corpus.json`: baseline-only calibration for all 22 parameterized TPC-H templates.
- `tpch-parameterized-rule-corpus.json`: rule-candidate search with 70 search parameter sets and 30 held-out parameter sets per query.
- `tpch-parameterized-local-llm-corpus.json`: local LLM candidate search using the same parameter split.
- `real-world/real-world-baseline-corpus.json`: baseline-only calibration for ten custom real-world anti-pattern scenarios over TPC-H data.
- `real-world/real-world-rule-corpus.json`: rule-candidate search for the same scenarios.
- `real-world/real-world-local-llm-corpus.json`: local LLM candidate search for the same scenarios.
- `real-world/real-world-mixed-corpus.json`: mixed selector using current static-rule coverage for RW-02/RW-03 and local LLM candidates for the remaining scenarios.

The all-query LLM manifest is generated by `tpch-generator/generate-duckdb-tpch.py` when the Compose `tpch-generator` service creates or refreshes `benchmark-data/tpch/sf<scale-factor>/`. The generator writes it next to the baseline manifest after loading SQL from DuckDB's `tpch_queries()` function.

The all-query LLM manifest uses `first-llm-candidate` as the expected candidate selector. The workload runner resolves this to the first returned LLM candidate for the configured model, so the manifest is deterministic and does not need to be regenerated when the Ollama model tag changes.

For full-corpus TPC-H LLM runs, the runner records one `workload_case_results` row per manifest entry and always continues after per-query outcomes. This is required for reportable analysis because no-candidate, same-as-original, validation-failed, and benchmark-failed cases remain visible in exported metadata instead of disappearing into console logs.

Use `scripts/run-main-run.ps1 -Model <tag> -Corpus tpch` for reportable local TPC-H runs. It selects the parameterized corpus through private helpers under `scripts/lib/`, enables held-out evaluation, requires 30 promotion pairs, captures EXPLAIN plan artifacts, exports metadata CSV files, writes a run manifest, and runs post-hoc analysis.

Use `scripts/run-main-run.ps1 -Model <tag> -Corpus real-world` for the custom real-world anti-pattern corpus. It reuses the generated TPC-H PostgreSQL database and points the same workload runner at the generated `real-world/` query, parameter, and manifest files.

Use `scripts/run-main-run.ps1 -Model <tag> -Corpus job-imdb` for the optional JOB/IMDB corpus after the external SQL and data files have been staged. The JOB/IMDB preparation helpers are private implementation scripts under `scripts/lib/prepare-job-imdb-resources.ps1` and `scripts/lib/prepare-job-imdb-workload.ps1`.

Use `scripts/run-fast-check.ps1` for the local regression gate covering semantic comparison edge
cases, declared parameter typing, coordinator-side typed candidate preflight before benchmark
execution, generator compilation, and PowerShell syntax checks.
